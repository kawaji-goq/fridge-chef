# Fridge Chef 要望まとめ

> 仕様策定前の「やりたいこと」整理ドキュメント。決定事項ではなく、議論のたたき台。

## 1. プロダクトの目的

冷蔵庫の中身をもとに、毎日の献立を考えてくれるアプリ。
「今日何作ろう」を解消し、食材を無駄なく使い切れるようにする。

## 2. ターゲットユーザー

- 家でご飯を作る人（主婦・主夫・一人暮らしの自炊層など）
- まずは **日本人ユーザー** を対象にスタート
- 将来的には **グローバル展開** を視野に入れる（料理文化・単位系・言語の差分を後から差し込めるように設計）

## 3. 主要機能（やりたいこと）

### 3.1 献立提案
- 冷蔵庫・冷凍庫の在庫から、毎日の献立を提案
- **家庭の状況（人数・年齢構成など）に応じて材料の量を調整**
- 栄養バランスを考慮し、**同じものばかり出さない**（履歴を見て偏らないようにする）
- **カロリー計算** に対応
- **持病向けの食事**（減塩・糖質制限・腎臓病食 など）に対応

### 3.2 在庫管理（冷蔵庫・冷凍庫）
- 食材の種類・数量・単位を保存
  - 例：卵 4個、牛乳 500ml × 2パック
- **賞味期限・消費期限** を管理
- **冷凍庫の在庫** も別管理で記憶
- （要検討）期限が近いものを優先して使う献立提案

### 3.3 好み設定
- 好きな料理（個別レシピ）の登録
- 好きな**料理の系統**（和食・中華・イタリアン など）の登録
- （後述の追加検討事項で深掘り）

### 3.4 ユーザー管理
- 将来的にはユーザー管理を行う
- ただし **仕様固めの初期段階ではログインを実装しない**（煩わしさを避けるため）
- ログイン機能は後付けできる前提で設計しておく
  - 例：初期はローカル/匿名ID、後でアカウントに紐付け移行できる構造

### 3.5 プラットフォーム
- **スマホの画面が主**（モバイルファースト）
- まずはモバイル Web か、ネイティブ/PWA かは要検討

## 4. 追加で検討したい論点（提案）

以下は今回の要望から自然に派生する論点。要否を判断したい。

### 4.1 食材の入力体験
- 手入力だけだとスマホで面倒になりがち
- **バーコード読み取り** / **レシート OCR** / **写真からの食材認識** の検討
- 「よく使う食材」のクイック登録

### 4.2 在庫の自動減算
- 提案された献立を「作った」と記録したら、使った材料を自動で在庫から引く
- どこまで厳密にやるか（調味料は管理外にする等のルール）

### 4.3 アレルギー・食事制限
- アレルギー（卵・乳・小麦・そば・落花生・甲殻類 など）の登録
- グローバル展開を見据えると：
  - **宗教的な制限**（ハラル、コーシャ、ヒンドゥー教の牛肉禁忌 など）
  - **ベジタリアン / ヴィーガン / ペスカタリアン**
- 持病向けと近接する論点なので、設計時に統合して考える

### 4.4 家族メンバー単位の管理
- 家族それぞれの好み・アレルギー・持病・カロリー目標を別々に持てるか
- 1食で「子供向け」「大人向け」を出し分けるか、全員共通の献立にするか

### 4.5 調理の現実的な制約
- **調理時間**（平日は30分以内など）
- **調理難易度**
- **使える調理器具**（コンロ口数、電子レンジのみ、オーブン有無 など）
- **作り置き / 翌日へ持ち越し** 対応

### 4.6 買い物サポート
- 「あと◯◯があればこのレシピが作れる」の提案
- **不足食材の買い物リスト** 自動生成
- スーパーの特売情報連携は将来課題

### 4.7 履歴・お気に入り
- 提案された献立・実際に作った献立の履歴
- お気に入りレシピのブックマーク
- 「最近食べてないもの」を優先する仕組み（栄養偏り防止と連動）

### 4.8 朝昼晩・シーン別
- 朝食・昼食・夕食で提案内容を変えるか
- お弁当用、来客用などのシーン対応

### 4.9 季節・旬
- 旬の食材を優先する提案
- 季節の行事食（おせち、ひな祭り、クリスマス など）

### 4.10 予算
- 1食あたりの予算上限を設定できると嬉しい人もいる

### 4.11 グローバル化を見据えた設計の留意点
- 多言語対応（i18n）
- **単位系**（g / oz、ml / cup、°C / °F）
- 食材名の地域差（同じ食材でも国で呼び方が違う、入手性が違う）
- レシピ DB は地域ごとに持つか、共通＋地域フィルタか

### 4.12 データの持ち方・プライバシー
- ログイン前のデータは端末ローカル？クラウド匿名同期？
- アカウント作成時にデータを引き継げる導線
- 持病・アレルギーは機微情報なので扱いに注意

### 4.13 AI/LLM の使い方
- 採用方針：**ハイブリッド**（既存 DB ＋ ネット検索 ＋ AI 生成）
- 詳細は「5. 献立提案アーキテクチャ方針」を参照

## 5. 献立提案アーキテクチャ方針（決定事項）

### 5.1 基本スタンス
- 4 章の **追加検討事項は基本的に全て対応する方針**（MVP で全部やるかは別途仕分け）
- 献立の **基本設計（候補絞り込み）はシステム** が行い、**最終案の決定は Amazon の AI**（AWS Bedrock 想定）に委ねる
- レシピソースは **既存 DB を一次**、足りない場合に **ネット検索 or AI 生成** で候補を補う

### 5.2 提案フロー（パイプライン）

```
[1] 在庫・ユーザー属性の収集
      ├─ 冷蔵庫/冷凍庫の在庫（期限含む）
      ├─ ユーザー/家族設定（人数・年齢・アレルギー・持病・宗教/思想的制限）
      ├─ 好み（好きな料理・系統）
      ├─ 直近の食事履歴（偏り防止）
      ├─ 制約（調理時間・器具・予算・シーン：朝昼晩）
      └─ 季節/旬

         ↓

[2] システム側「基本設計」レイヤ（決定論的フィルタ＋スコアリング）
      ├─ ハード制約で足切り
      │    └─ アレルギー / 宗教制限 / 持病禁忌（例：腎臓病→高カリウム食材NG）
      ├─ 在庫マッチ度・期限優先度をスコア化
      └─ 栄養バランス / 過去履歴の偏り / カロリー目標を反映

         ↓

[3] 候補ソース
      ├─ (a) 既存レシピ DB から候補抽出  ← 一次ソース
      ├─ (b) DB に十分な候補が無い場合
      │     ├─ ネット検索（外部レシピサイト等）
      │     └─ AI（Bedrock）による新規レシピ生成
      └─ 出典・信頼度を候補にメタ情報として付与

         ↓

[4] 最終案決定レイヤ（Amazon の AI / Bedrock）
      ├─ 入力：絞り込まれた候補群 ＋ ユーザーコンテキスト
      ├─ 役割：複数候補から「今日のおすすめ」を選定・並び替え・理由付け
      └─ 出力：提案献立（理由・栄養概算・代替案）

         ↓

[5] ユーザーに提示 → 採用 → 在庫自動減算・履歴記録
```

### 5.3 レイヤごとの責務分担

| レイヤ | 担当 | 理由 |
|---|---|---|
| ハード制約フィルタ（アレルギー・持病・宗教） | **システム（決定論）** | 安全性に関わるため AI の創作余地を入れない |
| 栄養計算・カロリー集計 | **システム（決定論）** | 数値の正確性担保。AI に計算させない |
| 在庫マッチ・期限優先のスコアリング | **システム** | ルールで十分表現可能 |
| 候補のレシピ取得 | **DB → ネット検索 → AI 生成** の順 | コスト・速度・信頼性のバランス |
| 最終的な「今日これにしよう」の選定と理由付け | **Bedrock の AI** | ソフト要素（魅力度・組み合わせ）の判断が得意 |
| 新規レシピ生成（DB に無い場合のみ） | **Bedrock の AI** | DB 補完。生成結果は栄養再計算をシステム側で実施 |

### 5.4 検討が必要な論点

1. **Bedrock のモデル選定**
   - Claude（Anthropic）を Bedrock 経由で使うのが第一候補
   - コスト・レイテンシ・日本語品質で比較
2. **ネット検索の実現方式**
   - 外部レシピサイトのスクレイピングは規約・著作権リスク
   - レシピ API（楽天レシピ API 等）の利用、または検索 API（Tavily / Brave 等）＋要約の方針
3. **既存レシピ DB の調達**
   - 自前で構築 / 公開データセット利用 / API 連携のどれを軸にするか
   - グローバル展開時の地域別 DB 戦略
4. **AI 生成レシピの安全性担保**
   - 栄養計算はシステム側で再計算（AI 数値を信頼しない）
   - アレルギー・持病禁忌食材を含む生成結果はシステム側で再フィルタ
5. **コスト設計**
   - 1 ユーザーあたり 1 日何回 AI を呼ぶか
   - 候補絞り込みでトークン量を抑える（候補は最終段に渡す前に N 件まで圧縮）
6. **キャッシュ戦略**
   - 似た在庫・似た条件は AI を再呼び出ししない
   - 生成レシピは DB に逆流して資産化（次回以降は DB ヒットさせる）
