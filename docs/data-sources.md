# データソース選定（レシピDB / 栄養 / ネット検索）

> 2026-05 時点の調査メモ。料金・規約は変動するため、本実装前に最新版を要確認。

## 1. 結論（推奨）

| 領域 | 採用案 | 補足 |
|---|---|---|
| 既存レシピ DB | **自前 DB を段階的に構築**（楽天レシピ API のランキングクロール ＋ AI 生成レシピの逆流蓄積） | クックパッド商用 API は事実上閉じている。楽天 API も検索系は薄いため「種データ」と割り切る |
| 栄養計算データ | **日本食品標準成分表（八訂）増補2023** ＋ 将来 USDA FoodData Central | 文科省データは出典明記で商用 OK・無料。栄養はシステム側で必ず再計算 |
| ネット検索 | **Brave Search API を一次**、必要に応じて Tavily を併用 | コスト効率重視。LLM 用の高品質要約が必要なフローでだけ Tavily |
| グローバル展開（Phase2 以降） | **Spoonacular** を地域別データ供給に追加候補 | フリー → 有料の階段が緩やか |

理由とトレードオフは 2〜4 章。

## 2. 既存レシピ DB をどう調達するか

### 2.1 候補比較

| ソース | 商用可 | データ量・質 | 検索API | 評価 |
|---|---|---|---|---|
| **楽天レシピ API**（Rakuten Web Service） | ◯（無料・要 ApplicationId） | レシピ画像・材料・調理時間あり | ✕ キーワード検索なし。「カテゴリ別ランキング（4件まで）」のみ | 種データ用に◯、本番検索には不足 |
| クックパッド | ✕ 研究用提供のみ、商用は書面同意必須 | 国内最大級 | 公開 API なし | 実質使えない |
| みんなのきょうの料理 / DELISH KITCHEN / クラシル | API なし | 高品質だが入手手段なし | — | 検討対象外 |
| 公開データセット（Recipe1M+, RecipeNLG 等） | 条件付き（研究目的中心） | 英語中心 | — | グローバル Phase2 で再評価 |
| **Spoonacular** | ◯（$10〜/月） | 50万件以上、栄養付き、英語中心 | ◯ 強力 | グローバル展開 Phase2 の主候補 |
| **Edamam** | ◯（$49〜/月） | 200万件以上、英語中心 | ◯ | Spoonacular に対しコスト劣勢 |

### 2.2 採用方針：「自前 DB を育てる」三層構造

```
┌──────────────────────────────────────────────────────┐
│ Layer A: 種データ（Seed）                              │
│   楽天レシピ API のカテゴリ別ランキングを日次クロール  │
│   → 自前 DB に正規化保存（数千〜数万件規模）           │
├──────────────────────────────────────────────────────┤
│ Layer B: 生成データ（Generated）                       │
│   Bedrock が生成し、ユーザーが採用したレシピを DB へ   │
│   逆流。栄養計算・禁忌チェックは文科省成分表で再検証   │
├──────────────────────────────────────────────────────┤
│ Layer C: 外部補完（External）                          │
│   DB に該当が無いときだけネット検索（3章）or AI 生成   │
│   結果のうち良いものは Layer B として資産化            │
└──────────────────────────────────────────────────────┘
```

採用理由：
- クックパッドが閉じている以上、**国内に「丸ごと買えるレシピ DB」は無い**前提で設計する必要がある
- 楽天 API はキーワード検索が無いので「在庫から献立提案」の主検索源には弱い → **種データ供給** に役割を絞る
- 生成 AI を併用するなら、**生成結果を捨てずに蓄積する**だけで時間とともに DB が独自性を持つ
- グローバル展開時は Layer A を地域別 API（Spoonacular 等）に差し替えれば構造を変えずに拡張可能

### 2.3 楽天レシピ API の運用上の注意

- **1 リクエスト 1 秒以上の間隔**（規約）→ クロールはバックグラウンドジョブで日次
- ランキング上位中心になるので**ロングテールが弱い** → Layer B/C で補う前提
- **出典表示が必須**（楽天レシピへのリンク等）→ UI に組み込む

### 2.4 法務・規約での留意

- レシピ本文の**全文転載は避ける**。自前 DB には「材料・分量・手順の構造化データ」を保存し、出典 URL とクレジットを必ず持つ
- 他社サイトの**スクレイピングは原則やらない**（ToS 違反リスク）。検索結果からの転載も同様
- AI 生成レシピは**自社著作扱い**で蓄積可能、ただし学習元の権利には配慮

## 3. ネット検索の実現方式

### 3.1 候補比較（2026-05 時点）

| API | 料金（目安） | 特徴 | 向き不向き |
|---|---|---|---|
| **Brave Search API** | 約 $3 / 1,000 req（Base）、Pro $5 | 中立的なインデックス、無料枠あり、LLM コンテキスト API も提供 | コスト効率の一次検索に最適 |
| **Tavily** | 約 $8〜10 / 1,000 req | LLM 用にランク済みスニペット、引用付き | 高品質要約が必要な二次パスに |
| Serper（Google ラッパ） | 約 $1 / 1,000 req | Google 検索結果取得、コスト最安 | Google 依存・規約変動リスク |
| Google CSE | 100 req/日 無料、超過 $5/1,000 | 公式だが用途制限あり | 小規模 PoC 向け |
| Bing Web Search | 体系再編中 | 仕様変動リスク | 様子見 |

### 3.2 採用方針：Brave 一次 ＋ Tavily 二次

```
[在庫＋制約] → DB 検索（ヒット十分？）
                    │No
                    ▼
              Brave Search API
              （クエリ：在庫材料＋"レシピ"＋制約）
                    │
                    ▼
              上位 N 件の URL とスニペット取得
                    │
              ┌─────┴─────┐
              ▼             ▼
       スニペットで十分    本文要約が必要
              │             │（持病・栄養精度要件など）
              ▼             ▼
        Bedrock で          Tavily で高品質抽出
        候補化              → Bedrock で候補化
```

採用理由：
- **コスト主導**：Brave $3 vs Tavily $9 は 3 倍差。MVP で全リクエスト Tavily にするとスケールしない
- Brave 結果でも Bedrock 側で要約・整形すれば多くのユースケースは賄える
- Tavily は「持病対応のように間違えると危ない領域」だけ呼ぶ二段構え

### 3.3 ネット検索結果の扱い

- 取得した本文の**全文は保存しない**。タイトル・URL・短い要約（自社生成）のみ
- ユーザーに提示するときは**出典リンクを明示**
- 同じクエリは**キャッシュ**（数時間〜1日）してコスト削減

## 4. 栄養データソース

### 4.1 採用：日本食品標準成分表（八訂）増補2023

- 提供元：文部科学省
- ライセンス：出典明記で**商用利用可**・無料
- フォーマット：Excel / PDF ダウンロード
- 用途：
  - 食材→栄養素マスタとして取り込み
  - レシピの栄養値計算（AI 生成レシピも**必ずシステム側で再計算**）
  - 持病対応のフィルタ（高ナトリウム・高カリウム・糖質量など）

### 4.2 グローバル展開時の追加

- **USDA FoodData Central**（米国農務省、無料・API あり）を Phase2 で追加
- 食材マスタを**地域非依存の中間 ID** で持ち、地域別成分表を紐付ける構造にしておく

### 4.3 食品成分データベース（fooddb.mext.go.jp）について

- Web 検索 UI は公開されているが、**公式 API は提供されていない**
- → 文科省配布の Excel データを取り込んで自前 DB 化するのが正攻法

```
[3] 候補ソース
   ├─ (a) 自前レシピ DB（Layer A: 楽天種 + Layer B: 生成蓄積）   ← 2章
   ├─ (b1) Brave Search → スニペットで足りる候補                 ← 3章
   ├─ (b2) Tavily → 高精度要約が要る候補                        ← 3章
   └─ (b3) Bedrock 新規生成                                      ← requirements.md
      ※ 全候補は文科省成分表で栄養再計算してから [4] に渡す       ← 4章
```

## 6. 決定事項（2026-05-25）

| 論点 | 決定 | 含意 |
|---|---|---|
| MVP での自前レシピ DB 構築 | **しない** | 楽天 API 直叩き ＋ Bedrock 生成のみで MVP を回す。Layer A（自前 DB 化）は Phase2 へ |
| ネット検索構成 | **Brave Search API のみ** | Tavily は Phase2 で精度要件が顕在化したら追加検討 |
| グローバル展開での Spoonacular | **採用前提で設計** | 食材マスタは「地域非依存の中間 ID」を持ち、地域別成分表 / API を後付けできる構造で初期設計 |

### 6.1 決定に伴う MVP 構成

```
[在庫＋制約]
   │
   ▼
[ハード制約フィルタ] ← システム
   │
   ▼
[候補ソース]
   ├─ 楽天レシピ API（カテゴリ別ランキング）← Layer A 直叩き
   └─ Bedrock 新規生成                       ← Layer C
        ※ 生成物の栄養値は文科省成分表で再計算（システム側）
        ※ 採用ログだけは残し、Phase2 で Layer B 化できるよう備える
   │
   ▼
[Bedrock で最終案決定]
   │
   ▼
[ユーザー提示 → 採用 → 在庫減算 → 履歴記録]
```

### 6.2 Phase2 以降の追加要素（保留）

- Layer A の自前 DB 化（楽天 API クロール＋正規化）
- Brave Search API による Layer C 補完
- Tavily 併用（持病対応など精度要件のあるフロー）
- Spoonacular 連携によるグローバル展開

### 6.3 MVP でも今やっておくべき設計上の備え

採用は Phase2 でも、以下は**初期設計の時点で**織り込む：

- **食材マスタを地域非依存 ID 設計に**（Spoonacular 採用前提）
  - 例：`ingredient.canonical_id` を中心に、地域別の名称・成分表 ID を別テーブルで紐付け
- **Bedrock 採用ログの保存**（Layer B 化の素地）
  - 生成レシピ・採用フラグ・栄養計算結果を構造化して残す
- **栄養計算は最初からシステム側**で実装（文科省成分表取り込み）
  - AI の数値をそのまま使わないルールを最初から徹底
