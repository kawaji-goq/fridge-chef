# MVP ユースケース（素案）

> 「冷蔵庫の中身から今日の献立が決まる」最小ループを最初に検証する。
> 4 章の要望（全機能対応の方針）のうち、**最小で価値が成立する範囲**だけを MVP に切り出す。

## 0. MVP の North Star

> **ユーザーがアプリを開いて 30 秒以内に「今日これ作ろう」と決められる**

これが成立しないなら他の機能を増やしても無意味、というのが MVP 仕分けの基準。

---

## 1. MVP に含めるユースケース（5 本）

### UC-1: 在庫を登録・管理する
**ペルソナ**: 家でご飯を作る人（日本人、スマホメイン）

**ストーリー**:
> 買い物から帰ってきて、冷蔵庫に入れた食材をアプリに登録したい。
> 卵 4 個、牛乳 500ml ×2 のように数量と単位で持ち、賞味/消費期限も入れたい。
> 冷凍庫に入れたものは別管理にしたい。

**受け入れ条件**:
- 食材名・数量・単位・保管場所（冷蔵/冷凍/常温）・期限を登録できる
- 一覧で見られる、編集・削除できる
- 期限が近い/切れたものが視覚的にわかる
- 入力は手入力でOK（バーコード/OCR/画像認識は Phase2）

---

### UC-2: 在庫から今日の献立を提案してもらう
**ストーリー**:
> 「今日何作ろう」と思ったら、ボタン一つで在庫から作れる候補を 3 つ出してほしい。
> アレルギーや好みは反映してほしい。同じものばかり出さないでほしい。

**受け入れ条件**:
- ホーム画面に「今日の献立を提案」ボタン
- 出力：3 候補（料理名 / 使う在庫食材と必要量 / 不足材料 / 推定カロリー / 出典）
- ハード制約フィルタ（アレルギー）はシステム側で必ず効く
- 好み（好きな系統）はソフト制約として AI 最終案決定に渡す
- 履歴を見て直近 N 日と被らないように
- 候補ソース：楽天レシピ API ＋ Bedrock 生成（決定済み）
- 栄養値はシステム側で文科省成分表から再計算

**非機能**:
- 提案完了まで 10 秒以内が目標
- AI 呼び出しはユーザーあたり 1 日数回前提でキャッシュ設計

---

### UC-3: 提案を採用して在庫から減らす
**ストーリー**:
> 提案を見て「これにしよう」と決めたら、使った材料を在庫から自動で引いてほしい。
> 後で履歴として振り返れるとうれしい。

**受け入れ条件**:
- 「これにする」ボタンで採用
- 使用予定の在庫アイテムが自動で減算（数量を確認・調整できる UI）
- 履歴に記録（日付・料理名・使用食材・栄養値・採用ソース）
- Bedrock が生成したレシピは、採用時にメタ付きで保存（Phase2 で Layer B 化する素地）

---

### UC-4: 好み・アレルギー・家族人数を設定する
**ストーリー**:
> 初回にざっと設定すれば、以降は自分に合った提案が出てくるようにしたい。
> 後で変えられるようにもしたい。

**受け入れ条件**:
- 初回オンボーディング（スキップ可）で以下を設定：
  - 食べる人数（大人 / 子供 で人数）
  - アレルギー（卵・乳・小麦・そば・落花生・甲殻類 等のチェック）
  - 好きな料理系統（和・洋・中・エスニック など複数選択）
  - 嫌いな食材（任意のフリーテキスト/タグ）
- 設定画面からいつでも編集
- **ログインは無し**。端末ローカル ID で永続化（Phase2 でアカウント引き継ぎ）

**MVP で出すスコープ（簡素化）**:
- 持病対応 / カロリー目標 / 宗教制限 は **Phase2**
- 家族メンバー個別管理 は Phase2（人数だけ持つ）

---

### UC-5: 直近の献立履歴を見る
**ストーリー**:
> 「最近何作ったっけ」を振り返りたい。
> 偏らないように同じ料理は続けて出ないでほしい。

**受け入れ条件**:
- 履歴一覧（日付逆順、料理名、サムネ的なテキスト表現）
- 提案エンジンが履歴 N 日分を参照して同一/類似料理を避ける
- 履歴から「もう一度作る」アクション（在庫に必要分があれば再提案）

## 2. Phase2 以降に回す機能（仕分け）

要望リスト（requirements.md 4 章）のうち、MVP に**入れないもの**：

| 機能 | フェーズ | 理由 |
|---|---|---|
| 持病対応（減塩・糖質制限・腎臓病食 等） | Phase2 | 安全性担保のため事前検証が必要。栄養 DB ＋ 禁忌ルール整備後 |
| 詳細な栄養トラッキング（日次累計） | Phase2 | MVP は 1 食単位の概算のみ |
| 家族メンバー別管理（個別アレルギー/好み/カロリー） | Phase2 | データモデルの複雑度が跳ねる |
| カロリー目標 / 体重管理連携 | Phase2 | 別ジャンル機能 |
| 宗教制限・ベジタリアン/ヴィーガン | Phase2（グローバル展開と同時） | 国内 MVP では優先度低 |
| 調理時間 / 難易度 / 調理器具による絞り込み | Phase2 | UI 設定項目が増えるため後付け |
| 朝昼晩・シーン別（弁当/来客） | Phase2 | MVP は「夕食 1 回」を基本想定 |
| 季節 / 旬の優先 | Phase2 | プロンプトで軽く触れる程度 |
| 予算上限 | Phase2 | 価格データソースが別途必要 |
| バーコード / OCR / 画像認識 | Phase2 | 入力体験の高度化、まずは手入力で UX 検証 |
| 買い物リスト自動生成 | Phase2 | 「不足材料」表示は MVP に含む（リスト機能は Phase2） |
| ネット検索（Brave）による外部レシピ補完 | Phase2 | 楽天 API ＋ AI 生成で足りるか検証してから |
| 自前レシピ DB の構築（Layer A） | Phase2 | データソース選定で決定済み |
| 多言語 / グローバル展開 | Phase3 | まずは日本特化 |
| アカウント / ログイン | Phase2 | 端末ローカル ID で MVP は進める |

## 3. MVP の機能境界線（迷ったときの判断軸）

「これ入れる？」で迷ったら以下で判定：

| 入れる | 入れない |
|---|---|
| 0 章の North Star に直接効く | UX 改善・効率化に留まる |
| 入れないと提案の質が**致命的に**落ちる | 提案の質を**もう少し**良くする |
| データモデルを後から差し替えると痛い | 機能追加で済む |
| ユーザーの**安全**に関わる（アレルギー） | 利便性向上（買い物リスト等） |

例：
- アレルギー設定 → **入れる**（安全＋提案の質に致命的）
- 持病対応 → **入れない**（安全に関わるが MVP では未整備の方が安全）
- 調理時間制約 → **入れない**（あった方がいいが致命的ではない）
- 食材マスタの地域非依存 ID 設計 → **入れる**（後から差し替えると痛い）

## 4. MVP ユーザーフロー（画面遷移の骨格）

```
[初回起動]
    ↓
[オンボーディング: 人数/アレルギー/好み]  ← UC-4
    ↓
[ホーム画面]
    ├─[在庫一覧] ←─→ [食材追加/編集]      ← UC-1
    ├─[今日の献立を提案] → [候補3つ表示]   ← UC-2
    │                       ↓
    │                  [これにする] → [在庫減算確認] → [完了]  ← UC-3
    ├─[履歴]                                ← UC-5
    └─[設定]                                ← UC-4 再編集
```

## 5. 決定事項（2026-05-25）

| 論点 | 決定 | 含意 |
|---|---|---|
| MVP のリリース範囲 | **UC-1〜5 を全て採用** | 「在庫→提案→採用→学習」ループを丸ごと検証する |
| 不足材料の扱い | **提案画面に表示のみ** | 「あと◯◯が足りません」表示。買い物リスト化は Phase2 |
| AI 生成レシピの安全表記 | **出典ラベルを常表示**（`楽天` / `AI 生成`） | 各候補カードにバッジ。同意モーダルは出さず UX を妨げない |
| 匿名→アカウント引き継ぎ | **端末 ID 設計に紐付けキーを仕込む** | 全データを `anonymous_user_id` で紐付け、Phase2 でアカウントへマージ可能に |

### 5.1 出典ラベルの UI 案

候補カードの右上などに小さなバッジ：
- `楽天レシピ` … 楽天 API 由来。出典 URL クレジット付き
- `AI 生成` … Bedrock 生成。栄養値はシステム計算済みの注記
- （Phase2 で `自前 DB`, `Brave 検索` 等が増える）

### 5.2 匿名 ID の設計指針

- 端末初回起動時に **UUID** を生成、ローカルストレージに保存
- サーバ側は `users` テーブルを `anonymous_user_id` で持ち、`account_id` は NULL 可
- Phase2 のアカウント機能導入時：
  - 既存 `anonymous_user_id` をそのまま使い続け、`account_id` を後付け
  - 複数端末を 1 アカウントに束ねるマージ処理を後で実装
- 全テーブルの外部キーは `user_id`（= `anonymous_user_id` 互換）に統一
